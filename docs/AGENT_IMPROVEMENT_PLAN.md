# План улучшения pm_agent: «живой» PM-агент

> Составлено: 2026-06-09. Старт реализации: 2026-06-10. Демо: 2026-07-15.
> Цель: агент понимает тулзы → строит план → действует → рефлексирует «достиг ли цели» →
> говорит как живой человек-проджект, уместно уточняет и помнит, с кем говорит.

---

## 0. Контекст и диагноз

### 0.1 Почему агент «тупой и топорный» (аудит архитектуры)

Сейчас pm_agent — это **классификатор намерения → белый список тулзов → «стоп на первом успехе»**,
а не агент, который рассуждает о тулзах. Это диспетчер ритуалов.

| Проблема | Где в коде | Эффект |
|---|---|---|
| Промпт прямо запрещает диалог | `pm_agent.py:13` («Ты НЕ чат-бот», «Без вопросов») | Машинный тон, нет личности |
| `action_only=True` | `pm_agent.py:111` | Агент физически не отвечает текстом — только отчёт об операциях |
| Система глушит уточнения | `react.py:493` `_is_chatty_delegation` | Вопрос агента распознаётся как баг и затирается |
| Сырое подтверждение | `react.py:1344` | Пользователь видит `Agent wants to call 'tracker_create_issue' (risk=medium) with: {...}` |
| Нет typing/стриминга | telegram bridge | Тишина 10–30с на долгих операциях |
| temperature=0.1 | `pm_agent.py:142` | Ноль вариативности |
| Таксономия из 9 интентов = потолок | `stage_graph.py:42` | Аналитика негде не живёт |
| «Стоп на первом тулзе» | `stage_graph.py:143` `any_read_answer` | read→compute→read невозможен внутри стадии |
| Тулзы выдаёт стадия, а не цель | `allowed_tools` whitelist | Агент не выбирает тулзы под задачу |
| Нет явной цели в state | — | Рефлексии не с чем сверяться |
| Рефлексия механическая, про отчёт | `react.py:801,911` | Проверяет «тул отработал» / «честен ли отчёт», а не «достигнута ли цель» |

**Контрольный пример.** «Сколько story points у Коли?»:
роутер → QUERY → один read (`board_snapshot` даёт счётчики задач, не суммы SP; либо `find_issues`) →
`any_read_answer=True` → финализация. LLM **не получает второго хода** просуммировать SP.
Итог: агент структурно не может ответить — (а) нет тула агрегации, (б) стадия умирает после первого
чтения, (в) нет рефлексии «я вообще ответил про SP?».

### 0.2 Достоинства текущего дизайна (сохранить)

Детерминизм, заморозка стадии на ход, форсированные рёбра (`backlog_plan → apply`),
защита от двойного `create` и от `close` сразу после `create`. Это полезно для надёжности «ритуалов» —
не выбрасываем, превращаем в guardrails на мутации.

### 0.3 Информация о пользователе: собирается, но в агента не доходит

Регистрация + привязка создают полную цепочку идентичности:

| Сущность | Поля | Где |
|---|---|---|
| User | email, display_name, **role** (dev/admin/user), active | `models.py:71` |
| TeamMembership (мост к Трекеру) | **tracker_login**, tracker_uid, tracker_display_name, match_status, **default_board_id**, role, **settings_json** | `models.py:209` |
| TelegramUserLink | telegram_user ↔ user ↔ team | `models.py:480` |

Бридж резолвит membership (`telegram_bridge.py:549`), **но только для авторизации**.
`InvocationContext` (`invocation.py:23`) несёт лишь `actor_external_id / actor_display_name /
actor_username`. В рассуждение попадает строка `«Ivan Petrov: <текст>»`. То есть
`tracker_login`, `role`, `default_board_id`, `settings_json` **до агента не доходят**.

Следствия: «создай мне задачу» / «сколько у меня SP» не резолвятся; спрашивается «какая доска?»,
хотя `default_board_id` известен; автономия не зависит от роли; персонализации нет.

### 0.4 Как использовать информацию о пользователе

| Поле | Использование агентом | Эффект |
|---|---|---|
| `tracker_login` | Резолв 1-го лица «мне/я/мои» → логин говорящего | Меньше уточнений |
| `tracker_display_name` команды | Дизамбигуация имён ДО вопроса | Спрашиваем, только если правда два «Коли» |
| `default_board_id` | Дефолтная доска/очередь | Уходит вопрос «какая доска?» |
| `role` (dev/admin/user) | Role-aware Autonomy Gate | Безопасность + меньше трения для своих |
| `settings_json` | Тон / детальность / язык / нотификации | Живой голос под человека |
| `User.display_name` | Приветствие по имени | Нет обезлички |
| `actor_user_id` | Атрибуция в actions/traces, петля action_feedback | Аудит и обучение |
| персональная память (новая) | Помнит ник→человек, предпочтения, прошлый разговор | Эффект «как живой» |

152-ФЗ: храним минимум PII, `settings_json` без чувствительного, атрибуция user_id с TTL.

## 1. Целевой цикл хода (что строим)

```
сообщение пользователя
  → UNDERSTAND/GOAL   : LLM строит Goal{intent, entities, success_criteria, missing_info, stage_hint}
                        обогащённый идентичностью (tracker_login, default_board, role, settings)
  → CLARIFY GATE      : если missing_info не закрывается контекстом/памятью → один человеческий вопрос, пауза
  → PLAN              : упорядоченный план тулз-шагов под success_criteria (можно multi-read + compute)
  → ACT (ReAct)       : выполнение; whitelist стадий — guardrail только на мутации
  → REFLECT           : LLM-судья сверяет результат с success_criteria → {goal_met, missing, next_action}
                        не достигнуто и есть бюджет → назад в ACT; недостижимо → честно сказать / уточнить
  → RESPOND           : человеческий голос; что сделал, что предположил, что предлагаю дальше
```

## 2. Фазы (содержание)

**Фаза 0. Каркас Goal.** `Goal` поверх стадий; стадия → мягкий хинт; whitelist стадий → guardrail на
мутации; `success_criteria` в state как якорь рефлексии. Файлы: новый `core/goal.py`, правки `react.py`,
`turn_plan.py`.

**Фаза 1. Многошаговость + понимание тулзов (чинит SP-пример).** Убрать «стоп на первом read»
(`stage_graph.py:143`) → терминальность по достижению цели; тул агрегации (`tracker_aggregate` или суммы
SP в `board_snapshot`); пагинация (снять тихое усечение); знание тулзов перенести из FSM-промпта в
docstring-описания + few-shot «вопрос → план тулз».

**Фаза 2. Настоящая рефлексия.** Шаг `reflect()` против `success_criteria` (не против отчёта); петля
доделывания вместо грубого retry (`react.py:849`); при недостижимости — честно сказать чего не хватает.

**Фаза 3. Живой диалог и уточнения.** Демонтаж `_is_chatty_delegation` и пересмотр `action_only`;
clarify-гейт ПЕРЕД действием; человеческий голос финализации вместо `_build_action_report`;
temperature 0.4–0.6 на разговорном слое; право на встречный вопрос на action-стадиях.

**Фаза 4. Память и персонализация.** Таблицы `person_profiles` / `project_memory`; `settings_json` →
тон/детальность/язык; атрибуция `actor_user_id` в actions/traces; связь с `action_feedback`.

**Фаза 5. Тулзы доски.** Связи blocks/depends/relates; bulk-операции (`/bulkchange`); жизненный цикл
спринта (list/close/перенос хвостов); выставить `list_boards`/`list_sprints` тулзами (методы уже есть в
`tracker.py`).

**Фаза 6. Оценка.** Golden-set 30–50 запросов (вкл. нестандартные); метрики: goal_met-rate,
лишние/недостающие тулзы, доля уместных уточнений, латентность.

## 3. Датированный план (старт 2026-06-10)

Недельные спринты. Каждый оставляет агента в рабочем состоянии.

### Спринт 1 — Идентичность + быстрые UX-вины · 10–16 июня 2026
- Расширить `InvocationContext`: `actor_user_id, actor_tracker_login, actor_role, actor_default_board_id, actor_settings` (бридж уже резолвит membership — прокинуть, `telegram_bridge.py:585`).
- Резолв 1-го лица «мне/я/мои» в `assignee_resolver` и в Goal.
- Человеческие подтверждения вместо сырого `Agent wants to call ...` + `typing`-индикатор.
- **DoD:** «создай мне задачу» ставит на `tracker_login` говорящего без уточнений; подтверждение человечное.

### Спринт 2 — Ядро Goal + многошаговость + агрегация · 17–23 июня 2026
- Объект `Goal` поверх стадий (Фаза 0).
- Тул агрегации / суммы SP; снять «стоп на первом read»; пагинация.
- Дефолтная доска из `default_board_id`.
- **DoD:** «Сколько SP у Коли?» → план `find/aggregate → ответ числом`, без обрезки.

### Спринт 3 — Рефлексия + живой диалог + уточнения · 24–30 июня 2026
- Шаг `reflect()` против `success_criteria` + петля доделывания (Фаза 2).
- Демонтаж `_is_chatty_delegation`; clarify-гейт ПЕРЕД действием; дизамбигуация по членам команды.
- Живой голос финализации; temperature 0.4–0.6 на разговорном слое.
- Role-aware Autonomy Gate по `actor_role`.
- **DoD:** на неоднозначный запрос — один человеческий вопрос; рефлексия фиксирует goal_met; роль влияет на подтверждения.

### Спринт 4 — Память и персонализация · 1–7 июля 2026
- Таблицы `person_profiles` / `project_memory`; `settings_json` → тон/детальность/язык.
- Атрибуция `actor_user_id` в actions/traces; связь с `action_feedback`.
- **DoD:** повторный «Коля» не вызывает уточнения; ответы под предпочтения; в трейсе видно «кто просил».

### Спринт 5 — Тулзы доски + автотесты · 8–14 июля 2026
- Связи / bulk / жизненный цикл спринта (по необходимости); `list_boards`/`list_sprints` как тулзы.
- Golden-set + метрики (Фаза 6).
- **DoD:** регресс-прогон зелёный; нестандартные запросы из набора проходят.

### Демо / буфер · 15 июля 2026
Срез «вау»: нестандартный вопрос → агент знает, кто спросил, строит план, при нужде уточняет
по-человечески, рефлексирует и честно отвечает.

**Критический путь до демо-эффекта:** Спринты 1→2→3 (10–30 июня) — минимум «знает меня + думает +
уточняет + рефлексирует».
