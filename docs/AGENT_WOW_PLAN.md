## Текущая оценка и разрывы

Шкала «тупой → мега»: было ~2/10, сейчас ~6.5/10. Чтобы дойти до 9/10, закрываем:

1. Живой голос — только на QUERY; действия всё ещё робо-отчёт.
2. Качество не измерено — нет eval/golden-set (это сейчас важнее новых фич).
3. Рефлексия мелкая — retry-once, не итеративная самокоррекция.
4. Память/персонализация не задействованы.

## Правка A — Живой голос на ВСЕХ стадиях (доделать Фазу 3)

**Проблема.** `_READ_VOICE_STAGES = {QUERY}` (`react.py:607`). INTAKE/STATUS/TRANSITION/BOARD/REORG/
HYGIENE возвращают `_build_action_report` → «Создана DARKHORSE-7», «Закрыто задач: 3». Робот.

**Что делать.**
- Ввести «двухслойную» финализацию для action_only: детерминированный отчёт остаётся как
  **факт-якорь** (для аудита/трейса), но пользователю отдаётся **человеческая обёртка**:
  короткий LLM-проход «оформи результат словами для коллеги» поверх отчёта.
  Пример: отчёт `«Создана DARKHORSE-7, исполнитель Коля»` → reply
  «Готово — завёл задачу на Колю, DARKHORSE-7. Поставить оценку и дедлайн?».
- Альтернатива дешевле (без доп. LLM-вызова): шаблоны-вербализаторы на каждый тип отчёта.
  Рекомендация: LLM-обёртка только для INTAKE/STATUS/REORG (где уместен тон), шаблоны — для
  массовых/служебных (close_issues, move).
- **Подтверждение по-человечески:** заменить `Параметры: {dict}` (`react.py:1431`) на
  естественную фразу из аргументов: «Создаю "MCP корнер кейсы" на Колю, 5 SP, дедлайн 12.06. Ок?».

**Регресс-защита.** PROACTIVE и дневной дайджест (строится напрямую в `daily_digest`) НЕ трогаем.
Отчёт-факт сохраняется в трейсе → тесты на `_build_action_report` остаются валидны (проверяем
факт-слой, а не финальный текст). Добавить тесты на вербализацию.

**Срок: 16–22 июня 2026. DoD:** ответы на действия звучат как от человека; подтверждение — фразой,
не словарём; нечатовые отчёты не изменились.

## Правка B — Eval-harness + golden-set (Фаза 6) — ПРИОРИТЕТ

**Почему первым по важности.** Сейчас «мега» недоказуемо: `test_query_voice` проверяет проводку
в изоляции, а не что живой LLM реально посчитал «8 SP». Любой рефактор промпта — рулетка.

**Что делать.**
- Golden-set 40–60 реальных запросов с разметкой ожидаемого исхода: создание, статус-апдейт,
  оформление доски, и especially **нестандартные**: «сколько SP у Коли», «сравни загрузку Коли и
  Ромы», «кому перекинуть задачи», «что горит к пятнице».
- Прогон end-to-end через ReActRunner с мок-Трекером, реальный LLM на планирование/судью.
- Метрики: goal_met-rate (по судье + ручной разметке), точность извлечения `entities/metric`,
  доля лишних/недостающих tool-calls, доля уместных vs лишних уточнений, латентность, токены.
- Зафиксировать baseline сегодня, гонять на каждый PR в промпт/реасонинг.

**Срок: 23–29 июня 2026. DoD:** `make eval` даёт таблицу метрик; baseline зафиксирован; в CI.

## Правка C — Настоящая петля рефлексии (+ опциональный reasoning-шаг)

**Проблема.** `_goal_met` даёт вердикт, но цикл — «retry один раз → иначе уточнение»
(`react.py:942-994`). Нет «не достиг → переосмыслил план → дочитал → синтезировал».
Плюс tier-1 для QUERY закрывает цель на «есть данные», если планировщик не проставил
`entities.metric` — хрупкая единая точка.

**Что делать.**
- Заменить retry-once на **bounded reflect-loop** (до N=3 витков): при `met=NO/NEEDS_MORE`
  судья возвращает `next_action`/`missing` → раннер дополняет план шагом и продолжает, а не
  финализирует. Бюджет итераций общий с `max_iterations`.
- Для QUERY с метрикой — не закрывать на tier-1 вообще, всегда судья (см. Правка из
  `AGENT_FIX_PLAN_QUERY_VOICE.md`, ужесточить).
- **Опциональный reasoning-шаг** (см. раздел «Thinking» ниже): на сложных целях добавить
  явное поле `reasoning`/`plan` в ответ планировщика — scratchpad перед действием. Включать
  селективно (только multi-step/аналитика), чтобы не растить латентность на простых операциях.

**Срок: 30 июня – 6 июля 2026. DoD:** на «сравни загрузку и предложи» агент делает несколько
витков (читает обоих → считает → синтезирует совет), а не обрывается после первого read.

## Правка E — Нативный reasoning YandexGPT 5 (подтверждено: поддерживается)

**Факт.** YandexGPT 5 поддерживает режим рассуждений через параметр
`completionOptions.reasoningOptions.mode`: `DISABLED` (дефолт) / `ENABLED_HIDDEN`. Построен на
YandexGPT 5 Pro с RL/RLHF; по доке Yandex Cloud повышает точность на сложных многошаговых запросах.
**Нюанс 1:** `ENABLED_HIDDEN` — рассуждение **скрытое**, токены CoT наружу не отдаются (только
итоговый ответ становится точнее) → это буст качества, а не «видимый thinking» для UI.
**Нюанс 2:** наш клиент (`llm.py`) ходит в OpenAI-совместимый `/v1/responses` с телом
`model/temperature/max_output_tokens`, а `reasoningOptions` живёт в **нативном** completion API
(`completionOptions`). Значит нужно либо проверить проброс reasoning через responses-endpoint,
либо для reasoning-вызовов завести второй путь на нативный `/completion`.

**Что делать.**
- Добавить в `LLMClient.complete` опциональный флаг `reasoning: bool` → при `True` слать
  `reasoningOptions.mode = ENABLED_HIDDEN` (нативный API) или эквивалент в responses-теле, если
  поддерживается. Дефолт — выключено (обратная совместимость).
- Включать reasoning **селективно** (связка с Правкой C): только на целях, помеченных
  планировщиком как multi-step/analytic, и в Goal-judge. На create/close — НЕ включать
  (нулевой буст, рост латентности/токенов; латентность уже больное место — нет стриминга/typing).
- Поскольку CoT скрытый, для инспектируемости оставляем **архитектурный** reasoning
  (plan/act/reflect раздельными вызовами) как основу, а нативный reasoning — как усилитель
  качества внутри этих вызовов.
- **Замерить буст на golden-set (Правка B):** прогон одних и тех же нестандартных запросов
  с `reasoning on/off` → сравнить goal_met-rate, точность арифметики (суммы SP), латентность,
  токены. Решение «включать ли по умолчанию для analytic-целей» — по цифрам, не на вкус.

**Риски.** Доступность модели/квоты на Pro-reasoning; рост стоимости и времени ответа; режим
помечен как beta — поведение может меняться. Гейтить фиче-флагом, дефолт off.

**Срок: 30 июня – 6 июля 2026 (вместе с Правкой C). DoD:** фиче-флаг reasoning в `LLMClient`;
на golden-set измерен буст on/off; включён по умолчанию только для analytic-целей, если цифры
оправдывают латентность.

## Приложение: Thinking / reasoning у модели

**Есть ли сейчас?** Нет нативного «thinking». LLM-клиент (`llm.py`) шлёт в Yandex Cloud
OpenAI-compatible Responses API (`/v1/responses`) тело только с `model/temperature/
max_output_tokens` (`llm.py:168-171`) — никаких `reasoning`/`thinking`-полей. Модель в проде —
`yandexgpt` (+`yandexgpt-lite` в фолбэке).

**Но де-факто есть «externalized reasoning»** — рассуждение размазано по отдельным LLM-вызовам:
Goal-decompose (план: intent/entities/success_criteria/missing_info) → ReAct-цикл (tool→observe→
next) → Goal-judge (рефлексия). Это инспектируемая, тестируемая «цепочка мыслей вне модели».

**Возможен ли нативный thinking? Да — подтверждено.**
- YandexGPT 5 поддерживает режим рассуждений: `completionOptions.reasoningOptions.mode`
  (`DISABLED` / `ENABLED_HIDDEN`), на базе YandexGPT 5 Pro. CoT **скрытый** (наружу не отдаётся,
  только повышает точность). Детали внедрения и нюанс с нашим `/v1/responses`-клиентом — см.
  **Правку E** выше. Источники — внизу документа.
- Провайдеро-независимый путь (дополняет нативный): **prompt-level CoT/scratchpad** — поле
  `reasoning` в JSON-ответе планировщика/судьи. Работает на любой модели, ничего не ломает,
  и в отличие от `ENABLED_HIDDEN` — **инспектируемо** (видно в трейсе).

**Даст ли буст?**
- ДА — на нестандартных/многошаговых задачах (SP-суммы, сравнение загрузки, «кому перекинуть»):
  scratchpad не даёт модели прыгать к выводу, заставляет реально посчитать/синтезировать.
- Ограниченно — на простых операциях (create/close): буст близок к нулю, а латентность и токены
  растут. Латентность уже больное место (нет стриминга/typing).
- Вывод: **внедрять reasoning селективно** — только на целях, помеченных планировщиком как
  multi-step/analytic, и в Goal-judge. Это Правка C. Полноценный «thinking-mode» на каждый запрос
  не нужен и вреден для UX. Архитектурный reasoning (plan/act/reflect раздельными вызовами) для
  mid-tier модели надёжнее, чем доверять «внутреннему» thinking — оставляем его основой, а
  scratchpad добавляем точечно.
- Альтернатива с наибольшим потолком: роутить сложные аналитические цели на reasoning-способную
  модель (если доступна), а простые операции — на дешёвую. Замерить буст на golden-set (Правка B).
