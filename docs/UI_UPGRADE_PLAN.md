# План апгрейда UI: ролевой продукт «как соцсеть»

Статус: черновик плана. Язык интерфейса — **русский**. Дизайн — в текущей стилистике
(`apps/web-ui/src/styles/index.css`, тот же shell/sidebar/surface), расширяем, не переписываем.

Цель: из операторской консоли PM-агента сделать ролевой продукт с тремя «лицами»
(пользователь / тимлид / разработчик), личными страницами в духе соцсети, метриками
доски и здоровьем команды, управлением кронами и агентами, входом по Telegram-коду.

## 4. Метрики доски и индекс здоровья (новый модуль `core/metrics.py`)

Считаем из Tracker по доске/очереди команды. Период — конфигурируемое окно (по умолчанию 14 дней).

**Базовые метрики (на члена и на команду):**
- Throughput — закрыто задач за окно.
- Cycle time — медиана времени `in progress → done` (по changelog/датам статусов).
- Lead time — от создания до закрытия.
- WIP — открытых «в работе»; Aging WIP — сколько висят дольше порога.
- Overdue ratio — доля задач с дедлайном в прошлом и не закрытых.
- Reopened ratio — доля переоткрытых.
- Sprint completion — план vs факт (если есть спринты, `list_sprints`).
- Blocked / без оценки / без исполнителя — гигиена доски.

**Индекс здоровья 0–100** — взвешенная композиция нормированных под-баллов
(каждый под-балл 0–100, веса в конфиге, объясняем «почему столько» в UI):
```
health = Σ wᵢ · scoreᵢ ,  Σ wᵢ = 1
напр.: flow(cycle/lead) .25, predictability(sprint) .15, timeliness(overdue) .2,
       wip_health(aging) .15, hygiene(no-estimate/assignee/blocked) .15, throughput_trend .1
```
Отдаём не только число, но и разбивку под-баллов + 2–3 главных «что тянет вниз».

**Хранение:** v1 — считаем on-demand с кэшем (TTL). v2 — таблица `board_metrics_snapshot`
(team_id, board_id, ts, metrics_json) для трендов на графиках без дёрганья Tracker.

**Endpoints:**
- `GET /me/board` — задачи пользователя по доске.
- `GET /me/stats?window=14d` — личные метрики/ряды для графиков.
- `GET /teams/{id}/health` — командные метрики + health index + breakdown (роль lead/dev).
- `GET /teams/{id}/members` — участники.

### «Скрамик» — питомец с системой уровней

**Решение:** визуально сейчас заглушка («Скоро»), но **систему уровней закладываем сразу** —
модель данных, правила начисления XP и расчёт уровня/настроения считаются из уже имеющихся
данных (Tracker issues + `Action`), чтобы на момент включения у питомца была история, а не ноль.

**Две независимые шкалы (классический тамагочи):**
- **Уровень (level)** — постоянный прогресс, только растёт. Питомец «эволюционирует» по тиру уровня
  (визуальные стадии: яйцо → детёныш → подросток → взрослый → …). Считается из накопленного XP.
- **Настроение/здоровье (mood 0–100)** — текущее состояние, колеблется: падает от просрочек и
  простоя, растёт от активности. Это «тамагочи-петля», мотивирует не запускать доску.

**XP — за реальную PM-активность (привязка к работе, не абстракция):**
| Событие | XP | Источник данных |
|---|---|---|
| Закрыта задача | базово +10, ×(1 + story_points/нормировка) | Tracker: статус → done |
| Закрыто **в срок/до дедлайна** | бонус +5 | deadline vs дата закрытия |
| Стрик «нет просроченных N дней» | +N (кап) | overdue ratio = 0 |
| Активность за день (комменты/переходы) | +2 (кап/день) | `Action`/Tracker changelog |
| (для лида) рост health index команды | +бонус | `core/metrics.py` |

**Mood (пересчёт ежедневно):** `mood = clamp(base + activity_bonus − overdue_penalty − idle_penalty, 0, 100)`.
Просроченные задачи и долгий простой «расстраивают» Скрамика; закрытия и порядок на доске — «кормят».

**Уровни:** пороговая кривая по накопленному XP (мягко растущая), напр.
`xp_for_level(L) = round(50 · L^1.5)` — L1≈50, L2≈140, L3≈260, L5≈560, L10≈1580.
Тиры визуала привязываем к диапазонам уровней (каждые ~3–4 уровня — новая стадия).

**Модель (миграция):** таблица `pet_state(user_id PK/FK, xp int, level int, mood int,
streak_days int, last_recalc_at, evolution_tier, state_json)`. Пересчёт — в том же дейли-кроне,
что и метрики (идемпотентно по дате). XP начисляется из событий доски за период.

**MVP-объём:** таблица + расчётчик XP/level/mood + endpoint `GET /me/pet` отдаём сразу
(чтобы копились данные); UI показывает **заглушку-превью** «Скоро» с уже реальным уровнем/прогрессом
или полностью статичную — на выбор. Полная анимация/механика кормления — после MVP.

## 7. Per-agent тулы и severity (developer) — РЕАЛИЗОВАНО

**Решение (отклонение от плана):** вместо новой таблицы `agent_tool_configs`
переиспользуем существующий overlay-механизм — тот же путь, которым уже едет autonomy.
Никакой миграции и нового рантайм-чтения из БД.

- Хранение: `AgentInstance.overlay["tools"] = {tool_name: {enabled: bool, confirm: bool|null}}`.
  `confirm`: `true` — всегда подтверждать, `false` — авто, `null/absent` — по риску тула.
- Рантайм:
  - `build_effective_config` (core/effective_config.py) парсит `overlay["tools"]` →
    `RuntimeConfig.disabled_tools` + `RuntimeConfig.tool_confirm` (overlay > spec).
  - `react.py` фильтрует disabled-тулы из набора для LLM и в confirm-гейте применяет
    per-tool override поверх risk-логики.
- Источник списка тулов: тулы агента — это в т.ч. MCP-тулы Tracker (`core/tracker_mcp.py`,
  риск из `_RISK`), зарегистрированные в реестре. `orchestrator.agent_tools()` отдаёт их
  name/description/risk; platform-api `GET /agents/{name}/tools` проксирует.
- Endpoints (console-api):
  - `GET /agents/{name}/tools` — объявленные тулы + эффективное состояние (enabled/confirm).
  - `PATCH /agents/{name}/tools` — массовое обновление (пишет `overlay["tools"]`, хранит только не-дефолты).
  - `GET /users` — все люди для разработчика.

Это и закрывает «confirms сейчас выключены, но у тулов они должны быть» — управление per-tool,
severity берётся из риска MCP-тула, override включаемо в UI.

## 8. Профиль и «соцсеть»

**Модель (миграция):** новая `user_profiles` (или расширение `users`):
- публичные: `avatar_object_key`, `title/position`, `contacts_json` (telegram/email/phone — что разрешено показывать), `bio`.

- **Решение по хранилищу (как проще):** локальный том в `console-api` (статическая папка `/data/avatars`,

**Endpoints:**
- `GET /me/profile` — полный (публичные + приватные).
- `PATCH /me/profile` — редактирование своих полей и контактов (с флагами видимости).
- `POST /me/avatar` — загрузка фото (object store, как в meeting-capture, либо локальный том; отдаём через `GET /users/{id}/avatar`).
- `GET /users/{id}/profile` — **публичный** вид (только разрешённые контакты, без приватного).

**UI:** личная страница рендерится по-разному:
- «я» — вижу приватные поля + кнопки редактирования + аплоад фото + виджеты (Скрамик, ачивки).
- «другой» — только публичные контакты и инфо.
